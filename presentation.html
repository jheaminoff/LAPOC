The Problem

LA's planning maze, in one sentence:

Getting a permit in Los Angeles means stitching together case numbers from different systems, decoding jargon from multiple departments, and knowing where to even start — a process that stalls homeowners, slows developers, and makes simple questions expensive.

Homeowners

They hear "ADU" or "permit" and have no idea where to begin. APNs, case IDs, LADBS versus City Planning — it is a foreign language. Most give up and overpay for help they should not need.

Developers

Every project touches CEQA, TOC, zoning, and permit history. Each lives in a different database. There is no single place to check a parcel's status, timeline, or outstanding items.

Contractors

Pulling a permit means navigating LADBS inspection codes, PC branch assignments, TCO versus CofO. Get it wrong and the job stalls. The information exists but nobody surfaces it.

The Solution

What is LAPOC?

LAPOC is an assistant that answers LA planning and permitting questions in plain language. It looks up parcels, case statuses, and process steps through a single chat interface that adapts to the person asking.
LLM-powered agent Parcel lookup Case tracking Workflow guidance Speech input and output Persona-adaptive responses

Three modes — Homeowner, Developer, Contractor — each gets answers pitched at the right level. A homeowner hears plain language with defined terms. A developer gets CEQA thresholds and zoning code. A contractor gets LADBS office procedures and inspection timelines.

Why It Matters

Why should the public care?

Time

A question that used to take a phone call, a visit to a department, or digging through a portal now takes one sentence typed or spoken. Ask in your own words, get an answer immediately.

One front door

Parcel data from Planning. Case status from LADBS. Process guidance from city code. All of it lives in one chat interface. No more figuring out which department to call.

Jargon translated

CEQA, ZIMAS, LOD, TCO, CofO, PC branches. Every acronym gets explained in context. You do not need to become an expert in city codes just to ask a question about your property.

Accessible

Speech input and output built in. A 3D mascot provides visual feedback during audio responses. No login, no download, no training required. Open the page and start talking.

Technical Deep-Dive

System Architecture

React + Vite → FastAPI Backend → Azure OpenAI ↓ SQLite
Backend (Python FastAPI + SQLite)
  • ■ main.py — entry point: loads env vars first, creates DB tables, sets up CORS, mounts 4 routers
  • ■ routers/chat.py — POST /chat runs the agent loop, generates follow-up suggestions, and produces TTS summaries
  • ■ agent/agent.py — Azure OpenAI loop (max 6 turns) with tool dispatch and persona detection
  • ■ agent/tools.py — five tools: lookup_parcel(), get_case_detail(), get_workflow(), lookup_address(), check_adu_eligibility()
  • ■ agent/prompts/system_prompts.jinja — persona-specific prompts written as Jinja2 blocks
  • ■ database.py — SQLAlchemy with a CWD-relative SQLite file
  • ■ models.py — Plot, Case, WorkflowStep, WorkflowPersona ORM models
  • ■ seed/seed_data.py — destructive reseed: 10 parcels, 30 cases, 135 workflow steps
Frontend (React 18 + TypeScript + Vite)
  • ■ Chat.tsx — main page: owns all state, speech hooks, and a 30-second fetch timeout
  • ■ ChatWindow.tsx — renders message bubbles with markdown support and card extraction
  • ■ MapCard.tsx — Leaflet map component rendered from MAP: sentinel data
  • ■ ParcelCard / CaseDetailCard / WorkflowTimeline — card components driven by sentinel-prefixed data from the agent
  • ■ ChatMascot.tsx — 3D ram model (React Three Fiber) with jaw animation synced to speech visemes
  • ■ useSpeechToken + useSpeechRecognizer + useSpeechSynthesizer — Azure Speech SDK pipeline
  • ■ Vite proxy — routes /plots, /cases, /chat, /static to the backend in development
  • ■ CSS Modules — every component has a paired .module.css file, no Tailwind or inline styles

Structured text protocol: Agent tools return data with sentinel prefixes (PARCEL:, CASE DETAIL:, WORKFLOW:, MAP:). The frontend parses these into rich card components while keeping the conversational prose clean.

Technical Deep-Dive

Agent Loop and Data Flow

Agent Loop (per chat message)

1. run_agent() — main LLM tool-calling loop (max 6 iterations, tool_choice: "auto")
2. generate_suggestions() — second LLM call, only on the first user message; produces 4 follow-up prompts
3. generate_speech_keynotes() — third LLM call on every response; yields a 1-2 sentence spoken summary (40 words max, no markdown)

All three calls share the same Azure OpenAI deployment. Both suggestions and speech keynotes have fallback logic if the LLM response is malformed.

Data Flow

User query → Vite proxy → FastAPI POST /chat → agent loop with tool dispatch

Tools query SQLite (10 parcels, 30 cases, 20 process types, 99 persona guidance rows)

LLM response → sentinel-prefixed structured data plus conversational prose → frontend card parsers

Three LLM calls per request (main + suggestions + speech). A 30-second frontend timeout sets the upper bound on chain depth.

Persona Adaptation

The frontend always sends persona: "auto". The LLM detects the user's role from their phrasing and appends [PERSONA:resident|developer|contractor] on the first reply. The Python backend strips the tag before returning the response to the UI.

System prompts live in system_prompts.jinja as named blocks — one per persona plus a suggest block. Edit the templates, not the Python code. Workflow guidance also filters by persona to show relevant step descriptions.

Speech Pipeline

useSpeechToken() fetches a short-lived STS token from GET /speech-token, refreshing every 9 minutes. This drives useSpeechRecognizer() (continuous speech-to-text) and useSpeechSynthesizer() (text-to-speech, voice en-US-Aria:DragonHDLatestNeural, MP3 output).

Viseme events from the synthesizer feed useVisemeScheduler(), which drives the mascot's jaw animation via a currentVisemeId state variable. Greeting audio is cached in IndexedDB so repeat visits skip re-synthesis.
